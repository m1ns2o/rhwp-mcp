This report tracks the current file-based `rhwp-mcp` stdio server coverage. The server targets HWP/HWPX file sessions, not a live browser editor session. A requirement-by-requirement Korean completion audit is written to [`assets/rhwp-mcp-goal-completion-audit.ko.md`](assets/rhwp-mcp-goal-completion-audit.ko.md).

## Implementation Model

`rhwp-mcp` is not a from-scratch rewrite of the HWP/HWPX editing engine. It is a thin MCP adapter over the existing `rhwp` Rust core: the MCP layer handles stdio JSON-RPC, sessions, file-root guarding, tool schemas, validation, and response serialization, while document operations call the existing `HwpDocument`/`DocumentCore` APIs.

When representative MCP roundtrip tests expose a preservation or fidelity gap, the fix is made in the shared parser/model/serializer/builder path so normal `rhwp` users and the MCP server keep the same behavior. The MCP server does not maintain a separate editing implementation. Recent preservation fixes follow that model for HWPX section metadata and `colPr` multi-column settings such as `id`, schema enum tokens, `MIRROR` layout, boolean `sameSz`, and per-column `colSz` width/gap values. The latest public government-form smoke follows the same rule: query-based table-cell replacement now uses the same reflow/dirty path as direct cell editing, and the SVG-to-PNG capture path now includes installed Linux CJK fallback fonts so rendered text appears in pixel diffs.
