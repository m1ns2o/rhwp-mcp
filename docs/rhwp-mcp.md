# rhwp MCP Server

`rhwp-mcp` is a local stdio MCP server for file-based HWP/HWPX editing. It opens files into in-memory sessions, exposes document/query/edit tools, and saves back to HWP or HWPX.

## Architecture

`rhwp-mcp` is an adapter over the existing `rhwp` Rust core, not a separate
HWP/HWPX editing engine. The MCP layer handles stdio JSON-RPC, sessions, file
guards, tool schemas, validation, and response serialization, while edit/query
operations call the existing `HwpDocument`/`DocumentCore` APIs. When MCP
roundtrip tests expose a preservation gap, the fix is added narrowly to the
shared parser/model/serializer/builder path so the core and MCP server keep the
same behavior. Query-based single text replacement follows the same shared-core
path: it now uses the existing whole-document search surface for body text,
table cells, and text boxes, and reports `cellContext` when the first match is
nested.

## Integrated Editor AI Panel

`rhwp-studio` includes a right-side AI panel for document-aware chat. The panel
collects current document metadata, page information, selected text, and nearby
cursor text, then sends that context to OpenAI, Claude, or a custom compatible
endpoint. Provider settings support API-key credentials, bearer tokens, and an
optional project-owned OAuth/token broker URL that returns an `access_token`.
Direct client-side API keys are intended for local/development use; production
deployments should prefer a server-side proxy or broker so provider credentials
are not exposed in the browser.

When `rhwp-studio` is embedded through `@rhwp/editor`, parent pages can register
provider settings without direct DOM access:

```js
const editor = await createEditor("#editor");
await editor.configureAi({
  provider: "openai",
  authMode: "apiKey",
  model: "gpt-5.5",
  apiKey: OPENAI_API_KEY,
});
await editor.openAiPanel(true);
```

For OAuth/WIF-style bearer credentials, configure `oauthEndpoint`,
`oauthClientId`, and `oauthScope`, then call `editor.refreshAiOAuthToken()`.
The broker endpoint receives `{ provider, client_id, scope }` and returns
`{ access_token }` or `{ token }`.

## Shape Shadows and HWPX Effects

`rhwp_get_shape_properties` now also returns a `shadow` object for ordinary
drawing shapes/OLE/chart objects when HWPX `<hp:shadow>` metadata is present.
The legacy flat fields (`shadowType`, `shadowColor`, `shadowOffsetX`,
`shadowOffsetY`, `shadowAlpha`) remain available for compatibility.
Shape border and solid-fill colors are exposed as both raw ColorRef numbers and
CSS RGB aliases: `borderColorHex`, `fillBgColorHex`, and `fillPatColorHex`.
`rhwp_set_shape_properties` accepts those hex aliases and writes canonical RGB
colors in HWPX `lineShape`/`winBrush` XML while keeping `borderColor`,
`fillBgColor`, and `fillPatColor` as raw numeric compatibility fields.
`rhwp_set_shape_properties` accepts `props.shadow` with `type` or `typeName`,
`colorHex`, raw `color`/`shadowColor`, `offsetX`, `offsetY`, and `alpha`;
`shadow: null` clears the shadow. Use `colorHex` for CSS RGB input/output; raw
numeric color fields remain HWP ColorRef-compatible.
For ordinary drawing-shape HWPX effects, `rhwp_get_shape_properties` exposes
`effects.threeD`, `effects.shadow`, `effects.glow`, `effects.softEdge`,
`effects.reflection`, `effects.blur`, and `effects.fillOverlay` from raw
`<hp:effects>` children.
Effect color fields under `effects.shadow`, `effects.glow`, and
`effects.reflection` use the same `effectsColor`/`rgb` JSON shape as picture
effects and expose `colorHex` when RGB values are present. Setters accept
`color.colorHex` as an RGB shortcut, accept snake_case color attribute aliases
such as `color_type`/`scheme_idx`, and also accept direct `colorHex`/
`color_hex`/`rgb` fields on the effect object when no nested `color` or
`effectsColor` object is provided. `color.rawChildXml`/`color.raw_child_xml`
preserves unknown nested
`hp:effectsColor` child XML for ordinary drawing effects. `effects.shadow` supports `skew` and
`scale` child objects, and `effects.reflection` supports `skew`, `scale`,
`alpha`, and `pos` child objects. The same point/range values can be supplied
with flat aliases such as `skewX`, `skew_y`, `scaleX`, `scale_y`,
`alphaStart`, `alpha_end`, `posStart`, and `pos_end`; these aliases are also
published in `tools/list` schemas. `effects.blur` and `effects.fillOverlay`
support attribute-level fields such as `radius` and `blend`, and
`effects.fillOverlay.solidFill`/`solid_fill` supports attribute-level solid
fill XML such as `colorHex`/`color_hex` and nested `effectsColor` RGB data
through `color`/`effectsColor`; `solidFill.rawChildXml`/
`solid_fill.raw_child_xml` preserves nested solid-fill child XML.
For effect objects, including `effects.threeD`, `rawChildXml`/`raw_child_xml`
can carry unknown top-level effect child XML alongside semantic fields;
extraction returns unknown children in the same array so edited effects can
still preserve deeper HWPX substructures.
`rhwp_set_shape_properties` accepts those fields as attribute-level objects,
for example
`{"effects":{"threeD":{"depth":120,"bevel":{"type":"ANGLE"}},"shadow":{"radius":444,"color":{"colorHex":"#152B41"},"skew":{"x":5,"y":6}},"glow":{"radius":333,"color":{"color_type":"RGB","rgb":{"r":12,"g":34,"b":56}}},"reflection":{"radius":111,"color_hex":"#778899"}}}`.
The `tools/list` input schema for `rhwp_set_shape_properties` now exposes the
same ordinary drawing effect fields, raw XML fallbacks, and color aliases so
schema-driven MCP clients can discover `effects.shadow`, `effects.glow`,

`rawHwpxChildXml` array remains visible for diagnostics and exact fallback
editing.
Document templates expose and accept the same ordinary-shape effect object under
`drawing_style.effects`, so a template can regenerate HWPX 3D, shadow, glow,
soft-edge, reflection, blur, and fillOverlay effects without hand-authored raw XML. The
targeted `document_template_accepts_shape_drawing_style_effects` test covers
JSON template input, HWPX export, reparse, template re-extraction, and
regeneration.
For HWPX-only ordinary drawing-shape children that do not yet have semantic
fields, `rhwp_get_shape_properties` exposes `rawHwpxChildXml` and
`rhwp_set_shape_properties` accepts `props.rawHwpxChildXml` (or
`raw_hwpx_child_xml`/`shapeRawXml`) as a string, string array, or `null` to
clear the fragments. These fragments are written back unchanged on HWPX export
and reparsed into the property bag.
HWPX import accepts common shadow type spelling variants such as
`RIGHT_BOTTOM`, `rightBottom`, and `right-bottom`. The MCP setter accepts the
same variants and HWPX export writes canonical OWPML enum names and RGB CSS
colors.

## Build

```bash
cargo build --bin rhwp-mcp
```

For a distributable native binary:

```bash
cargo build --release --bin rhwp-mcp
```

For local PATH-style installation from this checkout:

```bash
cargo install --path . --bin rhwp-mcp
```

The crate manifest declares `rhwp-mcp` as a native-only binary target, so Cargo
packages and release builds include the stdio MCP server without a separate
MCP-specific source tree. Use an absolute path to the installed or release

By default, the server only allows file paths under its current working directory. Set `RHWP_MCP_ROOT` to use a dedicated document root:

```bash
RHWP_MCP_ROOT=/path/to/hwp-files target/debug/rhwp-mcp
```

Writes are atomic. Existing files are replaced only when the tool call includes `"overwrite": true`.

HWPX input defaults to HWP output on save. Pass `"format": "hwpx"` explicitly to export HWPX.

## Tool Schemas

`tools/list` returns argument-specific JSON Schema for every registered tool. Most tools reject unknown arguments at the schema level. Alternative input groups such as `field_id` vs `name`, `image_base64` vs `image_path`, compare target session vs file, ingest/report/template object vs JSON string vs file path, and replace-by-query vs replace-by-position are exposed with `oneOf` and are also checked at runtime. Formatting and property-bag tools such as `rhwp_set_picture_properties`, `rhwp_set_shape_properties`, `rhwp_set_page_def`, and `rhwp_set_section_def` intentionally allow extra properties so MCP clients can pass the underlying `DocumentCore` formatter options. Style create/update `char_format`/`para_format`, direct apply-format `props`, `rhwp_set_page_def.props`, and `rhwp_set_section_def.props` still allow extensions, but their known fields publish the same type, range, enum, color-pattern, fixed-array, `tabStops`, page size/margin/binding, sectionId, section textDirection three-value enum/grid/wonggojiFormat/tabStopVal/tabStopUnit/outlineShapeIDRef/memoShapeIDRef/textVerticalWidthHead/numbering/start-parity/hide-flag/visibilityBorder/visibilityFill/first-page page-number and empty-line hiding/line-numbering constraints, and camelCase plus snake_case aliases enforced by runtime validation or the underlying `DocumentCore` setter.

`rhwp_get_style_list` returns compact style references by default. Pass
`"include_formats": true` to include resolved `charFormat` and `paraFormat`
summaries for each style, including semantic fields such as `fontSize`, `bold`,
`alignment`, `lineSpacing`, `indent`, shape-family ids, and raw-preservation
flags. Pass `"include_raw": true` (`includeRaw`) to include HWP-origin STYLE
record payloads as `rawHwpStyleBase64`/`raw_hwp_style_base64`; the response
also includes `langId`.

patches. Explicit `char_shape_id`/`charShapeId`,
`para_shape_id`/`paraShapeId`, `type`/`style_type`, and

`0` for paragraph style and `1` for character style.

or reuse document shapes, so a rejected style create call does not leave an
extra style, CharShape, or ParaShape in the session.

JSON object fields before shape creation. For example, numeric fields such as
`fontSize`/`lineSpacing`, boolean fields such as `bold`, CSS color strings,

## Document Template Linked Images

`rhwp_new_from_document_template` accepts `linked_image_policy` for `picture.external_path` and drawing `fill.image.external_path` values:

- `preserve_links` (default): keep linked images as Link BinData.
- `embed_accessible`: read linked image files under `RHWP_MCP_ROOT`/cwd, embed their bytes, and preserve links that cannot be read.
- `embed_required`: embed every linked image and fail the call if any linked image path cannot be read.

## Document Template Page Flow

`rhwp_new_from_document_template` accepts `page_flow_policy`:

- `preserve` (default): keep all extracted `break_before` page/column hints.
- `compact_artifacts`: remove empty page-break paragraphs, floating-object page breaks, bottom-anchored inline-picture page-break artifacts, and delayed sibling paragraph page-break artifacts before regeneration. This is useful for complex HWPX forms where structural template extraction can otherwise inflate regenerated page counts.

## Document Template Section Settings

`rhwp_extract_document_template` includes optional section-level `page_def`,
`section_def`, `page_border_fill`, `footnote_shape`, and `endnote_shape`
objects. Regeneration rebuilds multiple template sections as distinct
`Document.sections` and applies section-specific paper size, margins,
landscape/binding, page-number start settings, default tab spacing,
header/footer/empty-line hide flags, page-border basis, spacing, border lines,
fill, and note-shape fields such as number format, user/prefix/suffix
characters, start number, separator line metrics, numbering, placement,
superscript/inline flags, and raw unknown slots. Exact authored multi-section
pagination remains a follow-up item, but the current tests include an authored
3-section HWPX form (`samples/hwpx/hcar-001.hwpx`) with 17 tables and 242 table
cells through template extraction, regeneration, re-extraction, and HWPX
export/reparse. The representative MCP stdio smoke now also keeps that hcar
template at page count `6 -> 6` and matches regenerated pages 1/2/3 to the same
source pages with render scores `0.9999`/`1.0000`/`0.9982` and page-local
overflow `0`.

## Document Template Master Pages

`rhwp_extract_document_template` also includes top-level `master_pages` entries.
Each entry records the target section, apply target, extension/overlap/replace
flags, HWPX page number, textWidth/textHeight/textRef/numRef metadata, and the
master-page content blocks. Regeneration recreates these as section master
pages, and HWP/HWPX export/reparse preserves the generated base and LAST_PAGE
extension master pages.

`shadowType`, `shadowColorHex`, `shadowOffsetX`, `shadowOffsetY`, and
`shadowAlpha`.
For HWPX-only authored shape effects that are not yet structured,
`drawing_style.raw_hwpx_child_xml`
preserves raw child XML through template extraction, HWPX export/reparse, and
template regeneration. Ordinary drawing-shape
3D/shadow/glow/softEdge/reflection/blur/fillOverlay effects now also have a semantic
template path through
`drawing_style.effects` and a property path through `rhwp_get_shape_properties`
and `rhwp_set_shape_properties`, including effect shadow
`color.colorHex`/`color.color_hex`/skew/scale and flat point aliases,
snake_case color aliases, `glow.color.rgb`, direct glow/reflection
`colorHex`/`color_hex`, soft-edge radius, reflection

blur radius, fillOverlay blend/solidFill, solidFill nested RGB color and raw
child XML, plus effect-level raw child XML including `threeD.rawChildXml`.
Effect color nodes under shadow/glow and solidFill nested `effectsColor` also
preserve `rawChildXml`/`raw_child_xml` fragments through template extraction and
MCP property editing.
Remaining
effect-specific shape XML continues to use raw child XML preservation until
dedicated fields are added.
Complete shape gradient stop arrays are written to HWPX section
XML as `hc:color position="..."` entries and are read back on HWPX import. For
image fill, `fill.image` can carry
`image_base64` for embedded payloads or `external_path` for linked payloads,
plus `extension` and fill-mode properties; HWPX export writes the image brush
`binaryItemIDRef` against the regenerated manifest entry and keeps linked
payloads external in `content.hpf`. Shape groups can carry `children`
containing nested object placeholders, nested ShapeGroup placeholders, or
picture blocks;
grouped picture children regenerate with embedded image data or linked paths,
and top-level plus nested group component extra payloads roundtrip through HWP
export/reparse.

returns both top-level and nested ShapeGroup entries. The nested entry's

to inspect that nested group, and child shape entries expose absolute
`group_child_path` arrays for direct property edit/delete/z-order calls.
Picture child entries expose the same target form for direct Picture property
get/set calls, including caption creation and HWPX save/reopen verification.
Generated group resize/ungroup tests verify that persisted group scale is
applied to child geometry after HWP save/reopen.
Table, picture, and object-placeholder blocks can also carry an optional
`caption` with caption direction, vertical alignment, width, spacing, max width,
include-margin flag, and nested caption blocks; template regeneration preserves
those captions through HWP and HWPX export/reparse.

object payloads; OLE BinData references may be rebased to the regenerated
document's storage IDs. It is structural, not a lossless clone format.

Render SVG:

```json
{
  "name": "rhwp_render_svg",
  "arguments": {
    "session_id": "rhwp-1",
    "page": 0
  }
}
```

The structured response includes `page`, `svg`, `layout_overflow_count`,
`layout_overflows`, and `document_layout_overflow_count`.
`layout_overflow_count`/`layout_overflows` are scoped to the rendered page,
while `document_layout_overflow_count` preserves the whole-document diagnostics
collected during pagination. Overflow entries identify the page/section/
paragraph, element type, measured bottom, column bottom, and overflow pixels.

Preview one page for client-side UI integration:

```json
{
  "name": "rhwp_preview_page",
  "arguments": {
    "session_id": "rhwp-1",
    "page": 0,
    "include_png": false
  }
}
```

The structured response combines `page_info`, source format, dirty state, raw
SVG, embeddable HTML, page text, page markdown, optional PNG base64, and the
same layout-overflow diagnostics as render tools. Use `include_svg`,
`include_html`, `include_text`, `include_markdown`, and `include_png` to trim
the payload for chat clients, remote previews, or visual-diff workflows.

Render PNG:

```json
{
  "name": "rhwp_render_png",
  "arguments": {
    "session_id": "rhwp-1",
    "page": 0
  }
}
```

The structured response includes `page`, `mime_type`, `extension`,
`png_base64`, `byte_length`, `width_px`, `height_px`, `renderer`,
`layout_overflow_count`, `layout_overflows`, and
`document_layout_overflow_count`. The page-scoped and whole-document overflow
fields use the same semantics as `rhwp_render_svg`. The default binary uses
the existing SVG render path plus a safe `resvg` rasterization step; external
file hrefs are not loaded.

Compare rendered PNG pixels:

```json
{
  "name": "rhwp_compare_render_png",
  "arguments": {
    "session_id": "rhwp-1",
    "other_session_id": "rhwp-2",
    "page": 0
  }
}
```

The structured response includes the rendered size for both sides,
`changed_pixels`, `changed_percent`, `mean_abs_diff`,
`max_channel_abs_diff`, `same_size`, page-scoped layout overflow diagnostics,
and `document_layout_overflow_count` for whole-document diagnostics collected
during the two renders. Use `left_page` and `right_page` when page numbers
differ.
